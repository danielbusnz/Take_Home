assistant
Two different things, both just TypeScript `interface`s. Easiest framing for you: a TS `interface` is basically a Rust `trait` or a typed contract. Two places we use that idea here.

**1. Shared type = the wire contract, defined once.**

Your backend sends JSON, your frontend receives it. Both sides need to agree on the shape. Instead of the backend inventing `{status, sessionId}` and the frontend independently guessing those field names, you define it *once* in a file both import:

```ts
// shared/types.ts
export type LoginResponse =
  | { status: "mfa_needed"; sessionId: string }
  | { status: "done"; documents: Document[] };

export type Document = { name: string; bytes: string }; // base64 PDF
```

Backend imports it to type its responses, frontend imports it to type its fetch. Change a field name and TypeScript screams on *both* sides instantly. That's the whole point: one source of truth, no drift. "Shared" just means it physically lives where both can import it (a `shared/` folder in the repo).

In Python you'd do this with a Pydantic model passed around. Same instinct.

**2. Carrier interface = the plug shape every carrier must fit.**

This is the Rust trait analogy exactly. You define what *any* carrier must be able to do:

```ts
// shared/carrier.ts
export interface Carrier {
  name: string;
  login(user: string, pass: string): Promise<{ mfaRequired: boolean }>;
  submitMfa(code: string): Promise<void>;
  fetchDocuments(): Promise<Document[]>;
}
```

Then `MockCarrier`, `MapfreCarrier`, `AllstateCarrier` each `implements Carrier`. The *insides* are totally different (mock returns fake data, MAPFRE drives a real browser), but they expose the identical methods.

Why this matters: your backend never writes `if (carrier === "mapfre")`. It holds a `Carrier` and calls `.login()`, `.submitMfa()`, `.fetchDocuments()`, blind to which one it is. Adding tomorrow's second carrier = write one new class that satisfies the interface, register it in a lookup map. Zero changes to the server logic.

```ts
const carriers: Record<string, () => Carrier> = {
  mock:   () => new MockCarrier(),
  mapfre: () => new MapfreCarrier(),
};
```

So the two work together: the **Carrier interface** is the internal plug for swapping portals, the **shared types** (`Document`, `LoginResponse`) are the data flowing across the wire and through those methods.

assistant
No, and this is a good instinct to check because it's the opposite situation from where JWT belongs.

A JWT is a *signed, self-contained* token that carries identity claims, so a **stateless** server can trust the client without looking anything up. You used exactly that on Peeky's GitHub sign-in. It shines when you deliberately *don't* want to keep server-side state.

Here it's flipped. We have no user accounts, nobody's authenticating *to our app*, there's no database. And we inherently *have* server state, the live paused browser sitting in memory. The `sessionId` isn't proving who anyone is. It's just a **lookup key** into the in-memory `Map<sessionId, browserSession>`. So it's a plain opaque random string:

```ts
import { randomUUID } from "node:crypto";
const sessionId = randomUUID();           // or crypto.randomBytes(16).toString("hex")
sessions.set(sessionId, { carrier, browser, state: "AWAITING_MFA" });
```

No signing, no claims, no secret key, no expiry encoded in a token. When `/mfa` arrives with that id, you `sessions.get(sessionId)` and resume.

Why not JWT here, concretely:
- Nothing to put in the claims. There's no user id, no roles, no identity.
- JWTs are designed to *avoid* a server-side lookup. But we *must* do a server-side lookup anyway, because the browser session can't be serialized into a token. So a JWT buys you nothing and adds signing complexity.
- Random opaque id is actually the more secure default for a short-lived server session: it's unguessable, holds zero info, and vanishes from memory when the flow ends.

Rule of thumb: JWT when the token *is* the state (stateless server). Opaque id when the state lives on the server and you just need a handle to it. This is firmly the second.

So the server hands back a random `sessionId`, not a JWT. That's the only thing it sends back besides the status and, at the end, the documents.

# Writing good comments

Every comment must add information a reader couldn't infer from the code itself. If a comment restates what the next line does, delete it.

## Style

Never use em dashes or en dashes in comments. Use periods, commas, colons, or restructure the sentence. Arrows (→) for cause-effect are fine. Hyphens in compound words ("fire-and-forget") are fine. Hyphens used as punctuation are not.

## The test

For every comment, ask: could a reader figure this out from the code? If yes, delete. If no, the comment must add one of:

- **Higher-level intent.** What the block does, at a higher abstraction than the code.
- **Lower-level precision.** Units, ranges, null behavior, boundary conditions, ownership.
- **Rationale.** WHY this design, this magic number, this approach.
- **Cross-module assumptions.** Dependencies invisible from this file alone.
- **Non-obvious invariants.** Properties that must hold but can't be read off the code.

## Two kinds of documentation. Keep them separate.

**Interface docs** (immediately before the declaration):

- The contract: a simple mental model for callers.
- Cover everything any caller might need.
- Use abstraction-level language.
- Never describe implementation details here.

**Implementation docs** (inside the function):

- Rationale for tricky logic, magic numbers, cross-module assumptions.
- For long methods, label major blocks at a higher level.
- For shorter methods, often no implementation docs are needed.

## Document for every variable, argument, return value

- What it is (only if the name doesn't make it obvious).
- Units (ms, bytes, count, percent?).
- Boundary conditions (does "end" include the last value or one past it?).
- Null behavior (allowed? meaning?).
- Memory ownership (who frees? who reads? who mutates?).
- Invariants that must hold.

## Patterns to delete on sight

- `// increments counter` before `counter += 1`.
- "Tuned empirically" with no rationale.
- Elaborations that restate the docstring summary at length.
- Implementation details inside interface docs.
- The same fact documented in two places (one will go stale).

## Cross-module concerns

When module A depends on module B in a non-obvious way, document the assumption once and reference it from the other side:

```
// See "Zombies" in design-notes.md
```

Document each invariant exactly ONCE.

## Workflow

- Write comments FIRST, before the function body. Helps clarify the interface and surfaces complexity early.
- Higher-level comments rot less than line-by-line implementation comments. Prefer them.
- Use the same name for the same thing across the file. Avoid reusing a name for two different things.

## Red flags

If any of these come up, the code is wrong, not the comments:

- Can't name a variable clearly → variable is doing too many things.
- Method docs need to describe every internal step → method is too complex.
- Interface docs need to be very long to be complete → interface needs simplification.

"In-memory" = the data lives in your program's RAM, as ordinary variables, while the process is running. Nothing more. `sessions` and `carriers` are just JavaScript `Map` objects held in the Node process's memory.

The contrast is with a **persistent store** (Postgres, Redis, a file on disk), where data lives outside your process and survives it.

The defining property, and the tradeoff:

- **Fast and simple.** Reading from a `Map` is instant, no network call, no query, no setup. That's why it's enough here.
- **It dies with the process.** Stop the server, redeploy, or crash → the `Map` is wiped. Every parked session is gone. There's no "load it back from disk" because it was never on disk.

For this app that's actually fine, and deliberate (it's a CLAUDE.md invariant). The data is *ephemeral* anyway: a session only needs to live for the few seconds between `/login` and `/mfa`. Nobody needs it tomorrow. Persisting it would be pointless complexity, and you literally *can't* persist a live browser handle to disk anyway.

The one consequence to know: if the server restarts while someone is mid-login, their parked session vanishes and they'd have to start over. For a take-home with low traffic, acceptable. A high-availability version would move sessions to Redis so any server instance could resume them, but that's a scale problem you don't have.

So: in-memory store = "a variable holding data, gone when the program stops." `sessions` (changing per-request data) and `carriers` (static config) are both just that.

```ts
() => new MockCarrier()
```

Every time you call this, you get a brand-new `MockCarrier`. Call it three times, you get three separate independent carriers.

**Why use one here instead of just storing the carrier directly?** Compare the two:

```ts
// WITHOUT factory: one shared instance, created once
const carriers = { mock: new MockCarrier() };   // single carrier, made now
// everyone who logs in shares THIS ONE carrier

// WITH factory: a recipe, runs per login
const carriers = { mock: () => new MockCarrier() };  // a function
// each login calls make() and gets its OWN carrier
```

The first one is a problem: a carrier holds a live browser and per-session state (logged in or not, which page it's on). If two people log in at once and share a single instance, person B's login stomps person A's browser. They'd collide.

The factory fixes that: `make()` runs fresh on every `/login`, so each attempt gets its own isolated carrier + browser. No sharing, no collisions. The registry stores the *recipe* (the factory), and you cook a new one per request.

**The Rust/Python parallel:** it's like storing a constructor or a closure that returns `Self::new()` rather than storing one `Self`. In Python it'd be the difference between `{"mock": MockCarrier}` (storing the class, call `cls()` to instantiate) and `{"mock": MockCarrier()}` (storing one instance). The factory is the "store the class/constructor, instantiate per use" version.

So: **factory function = a function you call to get a fresh instance, used when each user needs their own copy rather than a shared one.**
