# Writing git commit messages

Default: one short subject line. State exactly what was done. Nothing else.

Add a body only when the WHY is non-obvious from the diff, and keep it to 1 or 2 terse lines. KISS.

Source: https://cbea.ms/git-commit/

## The seven rules

1. Separate subject from body with a blank line.
2. Limit subject to 50 chars (72 is the hard ceiling).
3. Capitalize the subject (override below for repo-specific prefix style).
4. No period at the end of the subject.
5. Imperative mood. "Fix bug" not "Fixed bug" or "Fixes bug".
6. Wrap the body at 72 chars.
7. Body explains what and why, not how. The diff already shows how.

## The subject-line test

A good subject completes the sentence: "If applied, this commit will ___."

Passes: "Refactor X module". Fails: "Fixed bug", "Sweeping changes", "More fixes".

## Style rules (hard)

- No em dashes or en dashes anywhere. Use periods, commas, or restructure.
- No semicolons. Comma or period.
- No emojis.
- No `Co-Authored-By:` lines.
- No `Generated with Claude Code` or similar footers.
- Never list Claude as author. Check `git config user.name` and `user.email` before committing.
- Don't repeat the prefix in the description. `fix: Fix X` is redundant.

## Plain verbs to default to

fix, add, remove, rename, move, extract, drop, revert, bump, pin, unpin, wire up, clean up, stub out, expose, hide, replace, restore, ignore, gate, skip.

## Banned AI-tell words

comprehensive, enhance, streamline, robust, leverage, utilize, ensure, rectify, "this commit introduces", "implement comprehensive", "this change adds support for".

If any appear in a draft, rewrite the subject.

## Repo-specific overrides

If the repo's `AGENTS.md` or `CONTRIBUTING.md` defines a style, follow that instead of rule 3:

- **Peeky style**: lowercase area prefix, colon, terse description. Examples: `tuning: zero post-release grace`, `demos: move examples/ to demos/`, `deny: ignore RUSTSEC-2026-0009`.
- **Conventional Commits style**: `feat(scope): ...`, `fix(scope): ...`, `chore(scope): ...` (used in Node/Angular/Vercel ecosystems).

The other six rules still apply.

## When a body is justified

- The change is non-obvious from the diff (workaround, perf trick, surprising ordering).
- The why needs context outside this file (incident ID, advisory, upstream bug).
- Closing a tracked issue: include `closes #N` (repeat per issue, a shared keyword only closes the first).

If the diff explains itself, write no body. One line is enough.

## Examples

Good, single-line:

```
tuning: zero post-release grace
demos: move examples/ to demos/
deny: ignore RUSTSEC-2026-0009 (time, blocked by tract pin)
fix rustfmt on screenshot guard in find_action
```

Good, with justified body:

```
routelet: cap the opt-in log and document data handling

Bound routelet_log.jsonl to its last 5000 lines. Storage cap, not a
privacy fix. Logging is already opt-in and redacted.
```

Bad (rewrite these):

```
Add comprehensive support for enhanced cursor positioning
```
Rewrite as: `cursor: snap to nearest target on partial match`.

```
This commit introduces a robust solution for the STT cutoff issue
```
Rewrite as: `stt: add 200ms post-release grace to capture trailing audio`.

## Workflow

1. Stage exact paths. Never `git add -A` or `git add .`.
2. `git status` to confirm only your changes are staged.
3. Draft the subject. Apply the "If applied" test.
4. Body only if justified above.
5. `git commit -m "..."` with HEREDOC for multi-line.
6. Never push unless asked.

## Red flags

- Subject describes how, not what or why.
- Subject would still make sense if pasted into a different PR. Too vague.
- Body restates the diff line by line.
- Two unrelated changes in one commit. Split them.
- Any banned word above.
