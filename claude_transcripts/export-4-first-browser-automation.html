Honest answer: your instinct isn't wrong that `fill()` is *less* human than typing, but it's optimizing the smallest lever while the big ones are already handled, and we have direct evidence it's fine here.

**What `fill()` actually does vs typing:** it sets the value and fires an `input` event, but it doesn't fire per-character `keydown`/`keyup`. So a script watching keystroke *cadence* sees a field fill instantly with no keys. That's a behavioral signal.

**But behavioral cadence is the weakest, last layer of bot detection.** The layers that actually gate a login are, in order of how much they matter: IP reputation, TLS/JA3 fingerprint, `navigator.webdriver` and other browser fingerprints, *then* behavior. The first three are exactly what **Browserbase's stealth + residential proxy handle for us**, and they're handled the same regardless of how we fill the form. Worrying about `fill` vs `type` is tuning layer 4 while Browserbase covers 1 through 3.

## Where Playwright's per-action time goes

**Source: playwright.dev/docs/actionability, browser-use.com/posts/playwright-to-cdp**
**Confidence: High**

Every `locator.fill()` does the following in sequence, each requiring at least one CDP round-trip:

2. Check visibility: the injected script queries bounding box and computed style to verify the element is non-empty-bounding-box and not `visibility:hidden`. Another in-browser JS evaluation.
3. Check enabled/editable: another JS evaluation checking `disabled` and `readonly`.
4. Scroll into view if needed (potentially `Runtime.callFunctionOn` + `Page.scrollTo`).
5. Execute the fill: focus the element, clear existing value, set `.value`, dispatch `input` event.

`locator.click()` does more: visibility, stability (waits for two consecutive identical bounding boxes across animation frames, so at minimum two round-trips), receives-events (checks hit-target at the center point), enabled. For Python/non-JS clients there is an additional architectural hop. The Browser Use blog (browser-use.com) documents it clearly: Playwright's Python (or any non-JS) client sends RPC over websocket to a Node.js relay server, which translates to CDP and sends it to the browser. With a remote Browserbase browser you add another network leg between you and Browserbase's server. So the round-trip cost per action is: local-to-Browserbase RTT + Playwright relay overhead. At 680ms per action that is plausible for a high-latency residential proxy path.

The Browserbase docs state explicitly: "each interaction with the page, even ones that seem atomic, can result in several underlying CDP commands, increasing the RTT." A single fill is 4-6 CDP messages. A click is 6-10.

## fill() options and alternatives

**Source: playwright.dev/docs/actionability, playwright.dev/docs/input, playwright.dev/docs/api/class-locator**
**Confidence: High**

### force: true on fill()

`locator.fill()` does not support `force: true`. Looking at the actionability table from the official docs, `locator.fill()` checks: Visible, Enabled, Editable. These three checks have no `force` bypass documented for fill. `force` exists on `click()`, `check()`, `tap()`, `hover()`, `dragTo()`, and similar pointer actions where "receives events" is the check being bypassed. Fill has no pointer-based check, so there is no `force` option surfaced in the fill API.

What you can do is set a very short `timeout` to fail fast rather than retry for 30 seconds, but that does not make the happy path faster.

### noWaitAfter on fill()

`noWaitAfter` on `locator.fill()` is documented but the behavior is nuanced. It was originally designed for actions that trigger navigation (click on a link, form submit) to avoid waiting for the navigation to complete. Fill does not trigger navigation by itself, so `noWaitAfter` on fill has essentially no effect on a normal input field. The option has been deprecated on `selectOption()` and is being phased out. For fill/fill/click, setting `noWaitAfter: true` on the fills saves nothing measurable, and on the final click it could cause problems if navigation begins before the next statement executes.

### locator.fill() vs locator.pressSequentially() vs page.fill()

- `locator.pressSequentially()` fires individual `keydown`/`keyup`/`keypress` events for each character. It performs zero actionability checks (the docs table shows no checks for pressSequentially). This sounds faster but it sends one CDP message per character. For a 20-character password that is 40-60 CDP messages versus 4-6 for fill. On a remote browser with 680ms RTT that is catastrophically slower. The official docs themselves say "you only need to type characters if there is special keyboard handling on the page."
- `page.fill()` is the legacy non-locator API. It does the same work as `locator.fill()` internally, just with a different selector resolution path. No meaningful speed difference.
- `elementHandle.fill()` is also effectively the same execution, just using a pre-resolved element reference. Minor theoretical saving on the locator-resolution step but the handle has its own staleness risk.

Conclusion: `locator.fill()` is already the fastest legitimate Playwright API for text input on a remote browser.

## CDP direct injection: Input.insertText, page.evaluate

**Source: stackoverflow.com/questions/23892547 (React nativeInputValueSetter pattern), transmitsecurity.com/blog/bot-detection-based-on-input-method-analysis, tykimos/cheliped-browser GitHub**
**Confidence: High on mechanics, Medium-High on bot detection risk**

### What happens under the hood

`page.evaluate()` executes JavaScript inside the browser process in one CDP round-trip (`Runtime.evaluate` or `Runtime.callFunctionOn`). Setting `element.value = 'text'` directly is one message. That is genuinely fewer round-trips than fill's 4-6 messages.

`page.evaluate(() =&gt; { el.value = 'foo'; el.dispatchEvent(new Event('input', {bubbles: true})); })` collapses the fill to a single CDP message round-trip.

### The React problem

React uses synthetic events and controlled inputs. When React renders `&lt;input value={state} onChange={handler} /&gt;`, the DOM `.value` property setter is wrapped. Setting `.value = 'foo'` directly on the element bypasses React's synthetic event system. The input event you dispatch manually has `isTrusted: false`. React's reconciler may not pick up the change, leaving the component state unchanged while the DOM shows the new value.

The correct approach for React controlled inputs, from the long-standing SO answer (stackoverflow.com/questions/23892547, 276k views, verified pattern used in testing tools since React 16):

```js
await page.evaluate((el, value) =&gt; {
  const nativeInputValueSetter = Object.getOwnPropertyDescriptor(
    window.HTMLInputElement.prototype, 'value'
  ).set;
  nativeInputValueSetter.call(el, value);
  el.dispatchEvent(new Event('input', { bubbles: true }));
}, emailEl, 'user@example.com');
```

This invokes the native setter that React's reconciler listens to, which causes React to detect the change and update its internal state. Without this, the `onChange` handler never fires, the form state stays empty, and the login POST contains empty credentials.

`Input.insertText` (raw CDP) bypasses all of this entirely and fires a trusted native input event. The cheliped-browser repo notes it uses `Input.insertText` for Korean IME support specifically where fill falls short. It is genuinely faster than fill and produces a trusted `isTrusted: true` event. However accessing raw CDP through Playwright requires `page.context().newCDPSession(page)` then `client.send('Input.insertText', {text: value})` after focusing the element, which still needs focus (another round-trip).

### Bot detection risk of evaluate-based filling

This is where Akamai matters. The Transmit Security research is explicit: bot detection systems examine the `isTrusted` property on DOM events. When a script dispatches an event, `isTrusted` is always `false`. Human keystrokes and paste events have `isTrusted: true`. Akamai's `sensor.js` collects keyboard events and their metadata. An `input` event with `isTrusted: false` on the email field is a flag. For a login form on a site running Akamai, `page.evaluate()` to fill fields is a meaningful bot signal. The VoidMob analysis of Akamai's sensor payload layer is clear: "Empty or synthetic payloads get scored as bot activity."

`locator.fill()` produces `isTrusted: true` events because Playwright drives them through the browser's actual input pipeline (`Input.insertText` or `Input.dispatchKeyEvent` under the hood, which the browser processes as trusted user input). This is a critical distinction.

**Verdict:** page.evaluate to fill is faster by roughly 3-4 round-trips per field, but on an Akamai-protected login it trades time savings for a real bot detection signal. Not worth it for this use case.

## Parallelizing the two fills on one page

**Source: playwright.dev/docs/actionability, browser-use.com/posts/playwright-to-cdp, Playwright architecture**
**Confidence: High**

Playwright's operations on a single page are serialized at the protocol level. The Node.js playwright server handles page operations through a single CDP session per page target. There is no public API to issue parallel CDP commands to the same page context. `Promise.all([locator1.fill(...), locator2.fill(...)])` does not actually parallelize the fills at the CDP level. Both promises start and the JS awaits them concurrently, but the underlying playwright RPC serializes them. You will observe the two fills executing one after the other, not simultaneously.

This is documented implicitly in the architecture: the Node.js relay maintains a single session per page. Browser Use's migration post explains this is one reason they dropped Playwright for raw CDP: "once you're doing heavy CDP-first work, you want finer-grained control and fewer round-trips."

Additionally, even if you could fire both fills simultaneously, they both need focus on the same DOM element at different times, creating a race condition on the `focus` event.

**Conclusion:** Parallelizing fills on one page is not possible with Playwright's public API and would not help anyway due to focus contention.

## Batching actions into fewer CDP calls

**Source: playwright.dev/docs/evaluating, browser-use.com/posts/playwright-to-cdp, Browserbase docs**
**Confidence: High on mechanics, Medium on practical savings**

Yes, `page.evaluate()` genuinely reduces round-trips. A single evaluate call is one CDP message regardless of how much JS runs inside it. You could fill both fields and simulate the events in one evaluate:

```js
await page.evaluate(([email, pass]) =&gt; {
  function reactFill(el, value) {
    const setter = Object.getOwnPropertyDescriptor(
      window.HTMLInputElement.prototype, 'value'
    ).set;
    setter.call(el, value);
    el.dispatchEvent(new Event('input', { bubbles: true }));
    el.dispatchEvent(new Event('change', { bubbles: true }));
  }
  const emailEl = document.querySelector('#email');
  const passEl  = document.querySelector('#password');
  reactFill(emailEl, email);
  reactFill(passEl, pass);
}, [email, password]);
```

This collapses 8-12 CDP round-trips (two fills) into one. Potential saving: 680ms × 1 to 1.5 = 680-1000ms in theory.

But as covered in question 3, this triggers `isTrusted: false` events. For Akamai, this is a meaningful bot signal on the login POST. On a non-Akamai form it is a legitimate optimization.

A safer batching approach is to pre-resolve selectors into element handles before the action sequence, then act on those handles, reducing locator resolution round-trips. But this saves maybe one CDP call per action, not the bulk of the time.

The Browserbase docs explicitly call out that "each interaction with the page, even atomic ones, results in several underlying CDP commands increasing RTT." Their own recommendation is geographic co-location (see question 6), not command batching.

## Disabling animations to cut stable-wait time

**Source: playwright.dev/docs/actionability, mergify.com/learn/flaky-tests/playwright, testdino.com/blog/playwright-flaky-tests**
**Confidence: High on mechanism, Medium on magnitude**

### The stability check

`locator.click()` requires the element to have "maintained the same bounding box for at least two consecutive animation frames." If your form has a CSS tab-switch animation (e.g. sliding tabs, fade-in), and the submit button or input fields are inside the animated element, Playwright will poll until the animation completes. Each poll is a CDP round-trip. If the animation runs for 200-300ms, Playwright may do 3-5 polled checks before proceeding.

`locator.fill()` does NOT check stability (per the actionability table). So tab animations do not block the fills. They only block the click if the button is inside an animating container.

### How to kill animations before the action

Inject CSS via `page.addStyleTag()` before interacting:

```js
await page.addStyleTag({
  content: `*, *::before, *::after {
    animation-duration: 0s !important;
    transition-duration: 0s !important;
  }`
});
```

Or use the Chromium launch flag `--force-prefers-reduced-motion` if you control the browser launch (not available via Browserbase's standard `connectOverCDP` path without their proxy config support).

The Mergify and TestDino flaky-test guides both document this as a standard reliability fix. For speed it specifically helps the `click()` call when the button is in an animated region.

**Expected saving:** If the tab animation was 300ms and the click was waiting it out, killing the animation could save 250-300ms from the click action alone. This is likely where a meaningful chunk of your click's 680ms is going.

**Bot detection risk:** None. You are modifying rendering behavior after page load, not the event dispatch pipeline. Akamai's sensor.js sees keyboard/mouse events, not CSS computed values. A page that loads instantly instead of with animations is not a bot signal.
